# Detection capabilities

Non-obvious implementation details for each detection feature. These are NOT discoverable from reading the code alone.

## Parser-level
- **HTML entry files**: `<script src="...">` (module and classic) and `<link rel="stylesheet" href="...">` / `<link rel="modulepreload" href="...">` create graph edges from HTML to referenced assets. Remote URLs skipped. HTML comments stripped before matching. HTML files exempt from unused-file detection.
- **JSX asset references**: lowercase intrinsic `<script src="...">` and `<link rel="stylesheet|modulepreload" href="...">` inside JSX/TSX files create `SideEffect` imports, mirroring the HTML parser. Only `JSXAttributeValue::StringLiteral` values are captured; expression containers (`href={expr}`) and capitalized React-style components (`<Script>`, `<Link>`) are ignored. Needed for SSR frameworks like Hono where layout components emit HTML via JSX. Combined with the web-root-relative resolver branch, `<link href="/static/style.css" />` correctly reaches sibling `static/style.css` files. See issue #105 (till's comment).
- **`html` tagged template asset references**: `` html`<script src="..."></script>` `` and `` html`<link rel="stylesheet|modulepreload" href="..."/>` `` inside any JS/TS file create `SideEffect` imports via the regex-based HTML asset scanner (`crate::html::collect_asset_refs`). Only the bare identifier tag `html` is matched — `css`, `sql`, `gql`, `styled.div`, and member/call expressions are deliberately ignored to avoid misreading unrelated tagged templates as HTML. Each template quasi is scanned independently, so an asset reference split across an interpolation boundary (`` html`<script src="${base}/app.js">` ``) is skipped rather than producing a garbled unresolvable specifier. Needed for SSR helpers like `hono/html`, `lit-html`, and `htm` where layout components emit HTML via tagged templates instead of JSX. See issue #105 (till's follow-up comment).
- **JSDoc `import()` type references**: JSDoc comments matching `import('./path').Member` inside `@param`, `@returns`, `@type`, `@typedef`, `@callback`, or other tag contexts are extracted as type-only imports. Supports single/double quotes, bare specifiers, parent-relative paths, union annotations with multiple `import()` expressions per comment, and nested member access (first segment wins). Side-effect-only `import('./types')` without `.Member` is recorded as `SideEffect` to keep the file reachable. Prevents false `unused-export` reports for types referenced only from JSDoc annotations in untyped JS files. Only `/** */` block comments are scanned (single-star `/* */` is not JSDoc). See issue #105 (till's comment).
- **Vue/Svelte SFC**: handles `>` in quoted attributes like `generic="T extends Foo<Bar>"`, `<script src="...">` external script support, HTML comment filtering. Template-visible import tracking: imports used only in markup (`{formatDate(x)}`, `utils.formatDate()`) are credited as used, preventing false unused-import/export reports. Vue credits only `<script setup>` bindings; Svelte excludes `context="module"` scripts from template visibility. Namespace member access in templates (`utils.formatDate`) tracked as member usage.
- **Vue/Svelte SFC `<style>` blocks**: `<style lang="scss">` and `<style lang="sass">` block bodies are scanned for `@import` / `@use` / `@forward` directives via the same regexes used for plain `.scss` files (issue #195 Case B). Plain `<style>` (no `lang`) and `<style lang="css">` are scanned for `@import` only. `<style src="./theme.scss">` external style references are extracted symmetrically with `<script src>` and emit a `SideEffect` import on the SFC. Each style-context import is tagged with `from_style: true` on `ImportInfo` so the resolver routes through the SCSS fallback chain (CSS-extension probe, partial convention `_filename`, include paths, node_modules) BEFORE the standard JS resolver. Without that routing, `./Foo` from a `Foo.vue` style block would resolve to the SFC itself (`.vue` precedes `.scss` in the resolver's extension list); the CSS-restricted probe selects the sibling `Foo.scss` instead. Languages other than CSS / SCSS / Sass (`postcss`, `less`, `stylus`) only have their `<style src>` reference seeded; their body syntax differs (e.g., Less `@import (reference)` modifier) and is out of scope.
- **Namespace destructuring**: `const { a, b } = ns` → member accesses. Rest patterns (`const { foo, ...rest } = ns`) → conservative whole-object use. Works with static/dynamic imports and require.
- **Unused import bindings**: via `oxc_semantic` scope-aware symbol analysis. Dead imports don't count as references, improving unused-export precision.
- **TypeScript overload dedup**: `export function foo(): void; export function foo(x: string): string;` treated as single export
- **Import-then-reexport pattern**: `import { X } from './a'; export { X };` is detected as a re-export (semantically equivalent to `export { X } from './a';`) instead of a local export. Without this, the bottom export collides with the original in duplicate-export detection and is never reached by re-export chain propagation. Handles renames on either side (`import { X as Foo }; export { Foo }` or `import { X }; export { X as Y }`), default imports (re-exported with `imported_name: "default"`), and mixed local + re-export in one statement. Order-sensitive: relies on imports appearing before exports in source order. Namespace imports (`import * as ns; export { ns }`) stay as local exports.
- **Class instance members**: `const svc = new MyService(); svc.greet()` and `this.service = new MyService(); this.service.doWork()` track the method as used. Chained `this.field` access uses synthetic `"this.field"` keys in `instance_binding_names`. Factory initializers through wrapper calls are also tracked: `const [svc] = useState(() => new MyService())` binds the first array-destructured element as an instance. Typed parameter / property chains through nested object types are also tracked: `constructor(private deps: { foo: FooClass }) { ... } this.deps.foo.foo()` registers `this.deps.foo -> FooClass` via `extract_nested_type_bindings` walking the `TSTypeLiteral`, then `static_member_object_name` records the full dotted-path access (`this.deps.foo`) which resolves through `binding_target_names` so `FooClass.foo` is credited. Same propagation runs through explicit `this.deps = deps` assignments via `copy_nested_binding_targets`, covering the no-accessibility-modifier form. Scope-unaware, false matches produce false negatives, not false positives.
- **Type-level member access**: `TSQualifiedName` (e.g., `type X = Status.Active`) tracked as member access. Mapped type constraints (`{ [K in Enum]: ... }`, `{ [K in keyof typeof Enum]: ... }`) and `Record<Enum, T>` mark all enum members as used via whole-object use.
- **TypeScript namespace exports**: `export namespace Foo { export function bar() {} }` extracts `Foo` as a single export with inner declarations as `NamespaceMember` entries, not as separate top-level exports. Runtime namespaces (no `declare`) are NOT type-only. `declare namespace`/`declare module` remain type-only. Nested namespaces flatten members into the outermost namespace.
- **Angular component metadata**: Full `@Component`/`@Directive` decorator metadata extraction prevents `unused-class-members` false positives. External templates (`templateUrl`) emit `SideEffect` imports and scan HTML for Angular syntax. Inline templates (`template:` string/template literals) are scanned in-place. `host:` object binding expressions extract referenced member identifiers. `inputs:`/`outputs:` string arrays mark listed members as framework-managed (handles `'name: alias'` syntax). Angular 17+ signal APIs (`input()`, `input.required()`, `output()`, `outputFromObservable()`, `model()`, `viewChild()`, `viewChildren()`, `contentChild()`, `contentChildren()`) set `has_decorator` on the property when the class has an Angular decorator. All template scanning supports `{{ }}`, `[prop]`, `(event)`, `[(ngModel)]`, `*ngIf`, `*ngFor`, `@if`, `@else if`, `@for`, `@switch`, `@defer (when expr)`, `@let name = expr;`, pipes. `*ngFor` and `@for` bindings are tracked as locals. `@let` variables are added as template-local bindings. `@defer` `when`/`prefetch when`/`hydrate when` conditions are scanned for member refs. **Inherited members and DI-injected service access** (issue #174): child-component template refs propagate up through `extends` chains so a member only defined on a base class is credited when referenced from a child's template. Member-access chains on unresolved identifiers (`dataService.getTotal()` in the template) are resolved through the importing component's typed instance bindings - constructor parameters with accessibility modifiers (`constructor(public dataService: DataService)`) and non-private typed property declarations are captured into `ClassHeritageInfo.instance_bindings` and used at analysis time to credit the target class's member as used. Inline templates reuse the visitor's existing `binding_target_names` resolver, external templates resolve in the analysis bridge.

## Resolution-level
- **Package.json `exports` subpath**: output dirs (dist/build/out/esm/cjs) mapped back to src/ with source extension fallback, including nested subdirs
- **Pnpm virtual store**: `.pnpm` paths mapped back to workspace source files. Handles injected deps, scoped/unscoped packages, peer dependency suffixes.
- **Package.json `imports` (`#subpath`)**: simple mappings, wildcard patterns, conditional exports. Per-package scoping.
- **Export / import condition names**: `oxc_resolver` is configured with the baseline condition set `["development", "import", "require", "default", "types", "node"]`. `"development"` ships unconditionally so packages that declare a `development` branch in `exports` / `imports` (common pattern: source files for dev, compiled output for prod; used by Vite, Vitest, esbuild, Rollup) resolve deterministically to source. When the React Native or Expo plugin is active, `"react-native"` and `"browser"` are prepended. Users extend the list via the `resolve.conditions` config key; those entries prepend ahead of everything else. See issue #135 and <https://nodejs.org/api/packages.html#community-conditions-definitions>.
- **React Native platform extensions**: `.web.ts`, `.ios.ts`, `.android.ts`, `.native.ts` resolved alongside standard extensions
- **Web-root-relative paths**: `/src/main.tsx` or `/static/style.css` resolved as a web convention (Vite/Parcel/Hono). Converts to `./src/main.tsx` and resolves from the source file's parent directory first, then falls back to project root. This handles monorepo workspace members where `site/index.html` referencing `/src/main.tsx` resolves to `site/src/main.tsx`, not `<root>/src/main.tsx`. Applied to HTML, JSX, TSX, and JS/TS source files (extensions `.html`, `.jsx`, `.tsx`, `.js`, `.ts`, `.mjs`, `.cjs`, `.mts`, `.cts`). JSX/TSX support is needed for SSR frameworks like Hono where layout components emit `<link href="/static/style.css" />` from JSX templates. See issue #105 (till's comment).
- **CSS scoped package imports**: CSS `@import` with scoped npm packages (`@fontsource/monaspace-neon/400.css`, `@company/design-system/styles.css`) are kept as bare specifiers for node_modules resolution. Unscoped bare imports with CSS/SCSS extensions are normalized to relative paths (`app.css` becomes `./app.css`).
- **Tsconfig path aliases**: per-file discovery resolves `@/utils` by finding nearest tsconfig.json per file
- **Broken tsconfig chain fallback**: when `resolve_file` fails with a tsconfig-loading error (broken `extends` chain, circular extend, self-referencing project reference, malformed JSON, IO error), fall back to `resolve(dir, specifier)` which bypasses `TsconfigDiscovery::Auto`. A solution-style `tsconfig.json` with a broken sibling reference no longer poisons relative/bare-import resolution for files covered by a healthy sibling. A single `tracing::warn!` per unique error surfaces the broken chain to the user. Path aliases in the broken workspace remain unresolvable (they were already broken in `tsc`). See issue #97.
- **SCSS partial resolution**: `@use 'variables'` and `@import '../styles/variables'` resolve to `_variables.scss` per SCSS convention. Extensionless bare specifiers in `.scss`/`.sass` files (excluding `sass:*` built-ins) are normalized to relative paths at extraction time. Resolution fallback tries: (1) `_filename` partial prefix, (2) `specifier/_index` directory index, (3) `specifier/index` plain index. Prevents false unresolved-imports, unlisted-dependencies, and unused-files from SCSS partial references.
- **SCSS include paths**: framework plugins contribute absolute directories to `PluginResult.scss_include_paths`. The Angular plugin reads `projects.*.architect.build.options.stylePreprocessorOptions.includePaths` from `angular.json`; the Nx plugin reads the same key from `project.json`. Nx workspace tokens (`{projectRoot}`, `{workspaceRoot}`) in include paths are expanded before resolution: `{projectRoot}` is computed from the config file's parent relative to the workspace root, `{workspaceRoot}` is stripped (paths are already resolved from workspace root). See issues #103, #114. Bare SCSS specifiers that fail the file-local `try_scss_partial_fallback` retry against each include dir, applying `<name>`, `_<name>`, `<name>/_index`, `<name>/index` candidates. Only triggers for `.scss`/`.sass` importers and normalized bare specifiers (those prefixed with `./` at extraction time) — explicit parent-relative paths like `../shared/vars` are left alone.
- **SCSS node_modules resolution**: bare SCSS/Sass specifiers that fail both the local-partial and includePaths fallbacks are retried against `node_modules/` by walking up from the importing file to the filesystem root (matching Node.js / Sass semantics). `try_scss_node_modules_fallback` probes `<nm>/<pkg>/<path>.{scss,sass,css}`, `<nm>/<pkg>/<parent>/_<name>.{scss,sass}` (partial convention), and `<nm>/<pkg>/<path>/{_index,index}.{scss,sass}` (directory index). The CSS-extension append handles Sass's fallback for files like `animate.css/animate.min` → `animate.css/animate.min.css`. Because `node_modules/` files are outside fallow's file index (excluded by default ignore patterns), candidates are probed via `is_file()` on disk and a successful hit returns `ResolveResult::NpmPackage` (extracted via `extract_package_name_from_node_modules_path`), keeping `unused-dependencies` / `unlisted-dependencies` accurate. Scoped packages (`@scope/pkg/path`) are not prefixed with `./` by `normalize_css_import_path`, so they flow through `oxc_resolver`'s bare-module path directly and never reach this fallback. See issue #125.
- **Entry point source fallback for ignored output dirs**: when `package.json` `main`/`module`/`exports` resolves inside an output directory (`dist`, `build`, `out`, `esm`, `cjs`) and the direct `src/` mirror in `try_output_to_source_path` does not exist, `resolve_entry_path` probes `src/index.*`, `src/main.*`, `index.*`, `main.*` in the package root. Prevents the entire `src/` tree from becoming unreachable in libraries whose compiled entry (e.g. `dist/esm2022/index.js`) has no one-to-one source mirror. Emits a single `tracing::info!` per fallback. See issue #102.
- **Workspace package self-reference & cross-workspace fallback**: when a bare specifier's package name matches a workspace package and normal resolution either fails or lands inside `node_modules` for that same package, `try_workspace_package_fallback` strips the package name prefix and resolves the subpath (`button`, `internal/base`, etc.) against a synthetic importer inside the workspace root. Uses `oxc_resolver`'s `resolve_file` so directory indices, source extensions, and the workspace's own `tsconfig.json` aliases still apply. Handles both Node.js v12+ self-referencing (`import '@org/pkg/sub'` from inside `@org/pkg`, common in Angular libraries built with `ng-packagr`) and cross-workspace imports in monorepos that haven't run `npm install`. See issue #106.

## Graph-level
- **Type-only cycle filtering**: `import type` edges carry `is_type_only` through `ImportedSymbol` to cycle detection. Edges where all symbols are type-only are excluded from Tarjan's SCC successor list, preventing false circular dependency reports from type-only bidirectional imports.
- **`export *` chain propagation**: multi-level barrel file chains fully resolved for transitive usage tracking
- **CSS Modules**: default imports (`import styles from '...'`) resolve `styles.className` to named exports via graph-level narrowing. Spread/`Object.values` handled conservatively.

## Analysis-level
- **Duplicate export common-importer filter**: duplicate exports are only reported when at least two files sharing the same export name also share a common importer in the module graph. Unrelated leaf files (e.g., SvelteKit route modules in different directories) that coincidentally export the same name are not flagged.
- **Decorated class members**: members with decorators (NestJS `@Get()`, Angular `@Input()`, TypeORM `@Column()`) not reported unused
- **Configurable class member allowlist**: the built-in Angular/React lifecycle check (see `predicates::lifecycle`) is extended at analysis time with two user-controlled sources: (1) top-level `FallowConfig.used_class_members`, (2) `Plugin::used_class_members()` / `ExternalPluginDef.used_class_members` on active plugins. Both sources feed `AggregatedPluginResult.used_class_members`, which `analyze/mod.rs` merges with the config list into a single `FxHashSet<&str>` passed to `find_unused_members`. Intended for third-party interface patterns where a library calls consumer methods reflectively (ag-Grid's `agInit`/`refresh`, Web Components' `connectedCallback`). The allowlist is scoped to class members — matching enum member names are still flagged. See issue #98.
- **Cross-package member access through barrels**: `import { Foo } from '@scope/lib'` followed by `Foo.bar` correctly credits `bar` on the originating file even when `@scope/lib`'s entry resolves to a barrel that re-exports `Foo` from a sibling module. Phase 4 chain resolution synthesizes a stub `ExportSymbol` on the barrel for reference tracking, so without further propagation the access map keys at the barrel and the detection loop's origin-keyed lookup misses where `members` are populated. `find_unused_members` walks each access key through `ReExportEdge` chains (named re-exports, renamed re-exports, and `export *` fan-out when no named edge matches) to every defining-site export and copies the access set. Cycle-protected. `export * as ns from './mod'` edges are intentionally skipped because two-level `ns.Foo.member` access isn't tracked at extract time. Same propagation runs for `whole_object_used_exports` (`Object.values(Foo)` on a re-exported enum). Covers both `unused_enum_members` and `unused_class_members`. See issue #178.
- **Inheritance-aware class members**: `extends` clauses are extracted to build an inheritance map. Parent class `this.*` accesses propagate to child class overrides (e.g., `BaseShape.describe()` calling `this.getArea()` credits `Circle.getArea()`). External member accesses propagate bidirectionally through the hierarchy. `super.method()` calls inside a subclass body are attributed to the parent via a `class_super_stack` in the visitor: the super-class identifier from the `extends` clause is pushed on class entry and popped on exit, and `Super.member` expressions record a `MemberAccess { object: <super_local>, member }` which `local_to_imported` then resolves to the parent's export. This covers the common pattern of a base class method only called via `super.method()` from subclasses. Only simple identifier extends are tracked (`class Foo extends Bar`), not mixin patterns. Methods called through parent-typed parameters remain unattributed (requires type inference, out of scope per ADR-001). See issue #130.
- **Private type leaks**: an exported value, function, or class whose public signature references a same-file type that is NOT exported by name is reported as `private-type-leak`. Detection runs in `find_private_type_leaks` (`crates/core/src/analyze/unused_exports.rs`) over `module_info.public_signature_type_references` (collected by the visitor's `SignatureTypeCollector`) cross-checked against `module_info.local_type_declarations`. Same-file private = the referenced name is in `local_type_declarations` AND not in the module's `exports`. References from ECMAScript private fields (`#field: T`) and TypeScript `private` accessibility members are NOT collected (the visitor short-circuits in `collect_class_signature_refs`), so they cannot leak. Qualified type names (`Foo.Bar.Baz`) reduce to the root identifier (`Foo`), since the leak is "the root namespace is private", not "every dotted segment is". Storybook story files (`*.stories.{ts,tsx,js,jsx,mts,cts,mjs,cjs}` and `*.story.*` variants) are skipped via `is_storybook_file`: the canonical `type Story = StoryObj<typeof X>; export const Default: Story = ...` pattern would otherwise generate one leak per story across thousands of UI library files. Framework routing convention files are also skipped via `is_route_convention_file` (Next.js App Router page/layout/template/loading/error/not-found/route/default/global-error/forbidden/unauthorized + the metadata files opengraph-image/twitter-image/icon/apple-icon/manifest/sitemap/robots; Next.js Pages Router; Gatsby `templates/`; Remix v2 + TanStack Router top-level + folder-route entries route/_layout/_index/index/__root; Expo Router `_layout` and `+*` files). All patterns prefixed with `**/` so they match in monorepo subpackages (`packages/web/src/app/.../page.tsx`). Real-world delta: amictus.ai 62 → 59 leaks (3 Next.js convention noise eliminated, 59 remaining are real React `Props`-pattern findings that warrant exporting the Props type). Companion logic in `suppress_signature_backing_types` removes any `unused_types` finding whose export is the type name backing a public signature in the same file, so exporting a backing type to fix one leak does not create a new "unused type" finding.
- **JSDoc visibility tags** (`@public`, `@internal`, `@beta`, `@alpha`, `@expected-unused`): exports annotated with any of these JSDoc/TSDoc tags are never reported as unused. Stored as a `VisibilityTag` enum on `ExportInfo` and `ExportSymbol`, preserving the semantic distinction (Public = API surface, Internal = sister packages/build tools, Alpha = early preview, Beta = unstable, ExpectedUnused = intentionally unused). Priority when multiple tags present: Public > Internal > Alpha > Beta > ExpectedUnused. Only `/** */` block JSDoc comments recognized. `@api public` TSDoc variant also supported. Word-boundary checked (e.g., `@internalizer` does not match). Survives incremental cache roundtrip via `u8` discriminant encoding. `@expected-unused` differs from other tags: when the tagged export IS actually used (has references from reachable modules), a stale suppression is reported instead of silently accepting it. This supports the pattern of marking API-completeness exports that become adopted over time.
- **Stale suppression detection**: after all detectors run, `SuppressionContext` compares consumed vs. all suppression comments. Unmatched `// fallow-ignore-next-line` and `// fallow-ignore-file` comments are reported as stale. Default severity: `Warn`. CLI-side suppression kinds (complexity, coverage-gaps, feature-flag, code-duplication) are excluded from stale detection to avoid false positives, since their suppression checks run in the CLI layer, not in `find_dead_code_full`. Suppressions for disabled rules (severity `off`) are correctly reported as stale since no detector checks them. The `Suppression` struct stores `comment_line: u32` alongside the target `line: u32` so stale reports point at the comment itself, not the suppressed line. `@expected-unused` JSDoc tags produce stale entries via the unused export detector when the tagged export becomes used. Both comment and JSDoc tag stale entries feed into the same `stale_suppressions` vec on `AnalysisResults` with a discriminated `SuppressionOrigin` enum.
- **Arrow-wrapped dynamic imports**: `React.lazy(() => import('./Foo'))`, `loadable(() => import('./Component'))`, `defineAsyncComponent(() => import('./View'))`, and any function call with an arrow or function expression argument whose body is a sole `import()` expression. The default export of the target module is credited as used (lazy loading wrappers always consume the default export). Handles expression body (`() => import('./x')`), block body (`() => { return import('./x'); }`), and function expressions (`function() { return import('./x'); }`). The import span is added to `handled_import_spans` to prevent duplicate side-effect edges from the generic `visit_import_expression` walker.
- **Dynamic import `.then()` callback**: `import('./lib').then(m => m.foo)` tracks `foo` as a named import from `./lib`. The standard Angular lazy-loading pattern (`loadComponent: () => import('./x').then(m => m.Component)`). Three sub-patterns: (1) expression body member access (`m => m.foo`) extracts specific member names as `destructured_names`, (2) destructured parameter (`({ foo, bar }) => { ... }`) extracts names from the object pattern, (3) identifier parameter with block body (`m => { ... }`) registers as a namespace binding for member-access narrowing. Also handles `({ default: m.Foo })` object literal bodies (React.lazy `.then` pattern) and function expression callbacks. The import span is added to `handled_import_spans` to prevent duplicate side-effect edges. See issue #115.
- **Infrastructure entry points**: Dockerfiles, Procfiles, fly.toml scanned for source file refs. Searches root and config/docker/deploy subdirs.
- **TypeScript project references**: tsconfig.json `references` field discovered as workspaces (additive with npm/pnpm workspaces)
- **CI file scanning**: `.gitlab-ci.yml` and `.github/workflows/*.yml` scanned for binary invocations (npx, direct binaries) AND positional file arguments (`node scripts/deploy.ts`, `--config foo.cjs`). Binary names map to npm packages (CI-only deps stay flagged-as-used); file-path arguments are seeded as project-relative entry patterns so scripts invoked only from CI are not reported as `unused-files` (issue #195 Case D). Line-based YAML parser; known limitations: variable interpolation (`${{ matrix.env }}/deploy.ts`), `\` line-continuations, and YAML anchors (`<<: *defaults`) are silently skipped.
- **Production mode**: excludes test/dev files, only start/build scripts, skips unused devDeps, detects type-only deps. Config file exclusion (`*.config.*`) is root-anchored only; nested `*.config.*` files (e.g., Angular's `src/app/app.config.ts`) are preserved. All production exclude globs use `literal_separator(true)` so `*` cannot cross path separators. See issue #111.
- **Test-only dependency detection**: `find_test_only_dependencies` identifies production dependencies imported exclusively from test/dev/config files (should be devDependencies). Uses a dual classification: glob patterns from `PRODUCTION_EXCLUDE_PATTERNS` for test/story files (`**/*.test.*`, `**/*.spec.*`, `**/__tests__/**`, etc.) and the curated `is_config_file()` predicate for tooling config files at any depth. The predicate approach avoids false matches on application config files like `app.config.ts` while correctly matching workspace-level configs like `packages/foo/vitest.config.ts`. See issues #111, #112.
- **Platform built-in modules**: `node:`, `bun:`, `cloudflare:`, `sass:` prefixed imports and Deno `std` are recognized as builtins — never flagged as unlisted dependencies. The `sass:` prefix covers the Sass compiler's built-in modules (`sass:math`, `sass:string`, `sass:color`, `sass:list`, `sass:map`, `sass:meta`, `sass:selector`) imported via SCSS `@use 'sass:math'`. See issue #104.
- **`ignoreDependencies` dual suppression**: listed packages are excluded from both unused dependency AND unlisted dependency detection, making it useful for runtime-provided packages (e.g., `bun:sqlite`).
- **@types/ unlisted dep suppression**: imports from `pkg` not flagged as unlisted when `@types/pkg` is in deps (TypeScript resolves types from `@types/` and erases the import regardless of `import type` syntax). Scoped packages use DefinitelyTyped convention (`@scope/pkg` → `@types/scope__pkg`).
- **Architecture boundary violations**: user-defined zones (glob patterns) with directional import rules. Fires at direct import site using resolved target's zone. Unzoned files are unrestricted in both directions. First-match zone classification. Self-imports always allowed. Inline and file-level suppression supported. Built-in presets: `layered` (presentation/application/domain/infrastructure), `hexagonal` (adapters/ports/domain), `feature-sliced` (app/pages/widgets/features/entities/shared), `bulletproof` (app/features/shared/server — shared zone covers components, hooks, lib, utils, utilities, providers, shared, types, styles, i18n). Presets auto-detect `rootDir` from tsconfig.json for pattern prefix. User zones/rules merge on top of presets (same-name replaces). Zones with zero file matches emit warnings.
- **Static test coverage gaps**: module graph reachability from test entry points to runtime files/exports. Not line-level coverage. Test files are identified by plugin patterns and production-mode exclusion rules. Reports runtime files with no test dependency path and runtime exports with no test-reachable reference chain. Gated by `--coverage-gaps` flag on the health command and `coverage-gaps` rule severity.
- **Complexity suppression**: health command complexity findings (functions and synthetic Angular template entries exceeding cyclomatic/cognitive thresholds) support inline suppression via `// fallow-ignore-next-line complexity` and `// fallow-ignore-file complexity`. The blanket `// fallow-ignore-next-line` (no kind) also suppresses complexity findings on the next line. The check happens in `collect_findings()` in `crates/cli/src/health/mod.rs` using `suppress::is_suppressed()` with `IssueKind::Complexity` (discriminant 18). File scores and vital signs are NOT affected by suppressions (they reflect actual complexity, not alerting). Synthetic Angular `<template>` findings in `.html` files get file-level suppression actions because line comments cannot be expressed in HTML; inline template findings in `.ts` files keep line-level suppression actions anchored at the decorator. See issue #108 (till's comment).
- **Synthetic Angular `<template>` complexity**: `health --complexity` emits synthetic `<template>` findings for Angular template control-flow density. External `templateUrl` files (`./foo.component.html`) are scanned at `parse_html_to_module_with_complexity` time and report against the `.html` file's coordinates. Inline `@Component({ template: \`...\` })` decorators are captured during the visitor pass (`ModuleInfoExtractor::inline_template_findings`) and synthesised after the visit by `parse::append_inline_template_complexity`, with the finding's line/col remapped onto the host `.ts` file's `@Component`/`@Directive` decorator start so jump-to-source lands on the decorator. This anchoring also lets `// fallow-ignore-next-line complexity` placed directly above the decorator suppress through the existing health-side check, with no special-case suppression plumbing. The CLI emits `suppress-line` actions for inline-template findings (matching JS/TS function complexity) instead of the HTML `suppress-file` action used for external templates. Template literals containing `${...}` expressions and `template:` properties bound to a variable are skipped (out of scope for the first cut). See issues #183, #187.
- **CRAP score threshold**: `--max-crap` (health and audit, default 30.0, `health.maxCrap` in config) compares each function's CRAP score (`CC^2 * (1 - cov/100)^3 + CC`) against the threshold. Per-function CRAP is produced as a side effect of file scoring (`compute_file_scores` in `scoring.rs`) and merged into `HealthFinding` by `merge_crap_findings` in `health/mod.rs`. Uses Istanbul per-function coverage when `--coverage` resolves; falls back to a graph-based estimator (directly test-referenced = 85%, indirectly reachable = 40%, untested = 0%). Findings triggered by CRAP alone, or alongside cyclomatic/cognitive, carry populated `crap` and (when Istanbul matched) `coverage_pct` fields and share the `complexity` suppression kind. See issue #141.
- **Script binary-to-package resolution**: npm `scripts` binary names are mapped to package names via a 4-step chain: (1) static `BINARY_TO_PACKAGE` table for known divergences (`tsc` -> `typescript`, `ng` -> `@angular/cli`, etc.), (2) `node_modules/.bin/<binary>` symlink target path extraction, (3) dynamic reverse map built from each dependency's `package.json` `bin` field (handles both string `"bin": "./cli.js"` and object `"bin": {"attw": "./cli.js"}` forms), (4) identity fallback (binary = package name). The dynamic map probes `node_modules/` at both the project root and each workspace root, handling non-hoisted setups (pnpm strict, Yarn workspaces). Built once per analysis run. See discussion #107. Positional file arguments (`tsc ./scripts/deploy.ts`, `node ./scripts/deploy.js`) extracted from script values are seeded as entry patterns to keep CI-only / deploy-only sources reachable. For workspace packages, paths are joined with the workspace's project-relative prefix and `..` segments are normalized via `normalize_script_entry_pattern`; `..` segments that escape the project root are skipped instead of producing unmatchable globs (issue #195 Case C).
- **Vite `css.preprocessorOptions.{scss,sass,less,stylus}.additionalData`**: the Vite plugin extracts the string value at each preprocessor's `additionalData` key from `vite.config.{ts,js}` and scans it for stylesheet imports via the same regex used by the CSS file parser. SCSS / Sass bodies seed discovered local style paths as entry patterns and use SCSS-aware normalization (extensionless local specifiers become `./X` partials); package imports such as `bootstrap` and `bootstrap/scss/functions` are credited as referenced dependencies instead of inert entry globs when no matching local style candidate exists. Less / Stylus strings are parsed for `@import "path"` and package dependency credit, but local `.less` / `.stylus` files are not currently part of the discovered source-file set. Function-form `additionalData: (source, file) => ...` is silently skipped (out of static-analysis scope). Specifiers are stripped of leading `./` before push because globset matches against project-relative paths without that prefix. See issue #195 Case A.
- **Cypress `e2e.specPattern` / `component.specPattern` / `supportFile`**: the Cypress plugin reads `cypress.config.{ts,js,mjs,cjs}` and seeds `e2e.specPattern`, `component.specPattern`, `e2e.supportFile`, and `component.supportFile` (string or array of glob patterns / paths) as entry patterns. Static patterns under `**/*.cy.{ts,tsx,js,jsx}`, `cypress/**/*.{ts,tsx,js,jsx}`, and `cypress/support/**/*.{ts,js}` seed Cypress's default component spec discovery and default Cypress 9-style layout; the config-driven extraction handles Cypress 10+ projects whose specs live outside `cypress/` (e.g. `tests/**/*.cy.ts`, `src/**/*.cy.ts`). See issue #195 Case E.
- **Script multiplexer handling**: `concurrently`, `npm-run-all`, `npm-run-all2`, `run-s`, `run-p`, `run-s2`, `run-p2` are recognized as script multiplexers. Their positional arguments are treated as script names (not binary names or file paths) and excluded from `used_packages`. Only the multiplexer binary itself is recorded. This prevents false "unused dependency" reports for packages referenced only via `npm:` prefixed script delegation (e.g., `concurrently "npm:dev" "npm:test"`).
- **Entry export validation** (`--include-entry-exports`): when enabled, entry file exports (package.json main/module/exports, framework convention files) are subject to unused-export detection instead of being automatically marked as used. Catches typos in framework exports (e.g., `meatdata` instead of `metadata` in Next.js pages). Still respects plugin `used_exports` rules for partial checking. CLI-only flag, not in config schema.
- **Single-file filtering** (`--file`): scopes check output to specified file(s). Full graph still builds (correct results require cross-file analysis), but only issues in matching files are reported. Dependency-level issues (unused deps, dev deps, type-only deps) are suppressed in file mode since they are project-wide properties. Warns when a path is not found. Useful for lint-staged pre-commit hooks.
- **Hotspot ownership signals** (`--ownership` on `health --hotspots`): per-file ownership metrics derived from `git log --use-mailmap --format='%at|%ae'`. Computed only when `--hotspots` runs and a file passes the `min_commits` threshold. Signals: bus factor (Avelino truck factor — minimum contributors covering ≥50% of recency-weighted commits), contributor count (after bot-pattern filtering), top contributor with stale-days, recent contributors (top-3), declared CODEOWNERS owner, drift (file age ≥30d AND original author share <10%), tristate `unowned` (true=no rule matches; false=rule matches; null=no CODEOWNERS file). Emitted via `OwnershipMetrics` on `HotspotEntry` (`skip_serializing_if = "Option::is_none"` for backwards compat). JSON injects ownership-aware actions: `low-bus-factor`, `unowned-hotspot` (with synthesized CODEOWNERS pattern), `ownership-drift`. Email privacy mode (`ownership.emailMode` config or `--ownership-emails` CLI): `handle` (default, local-part only with GitHub noreply unwrap), `hash` (`xxh3:` non-cryptographic pseudonym), `raw`. Bot patterns are globset matches against the raw author email; defaults cover `*\[bot\]*` (escaped brackets), `dependabot*`, `renovate*`, `github-actions*`, `svc-*`, `*-service-account*`. `*noreply*` is intentionally NOT a default because most human GitHub contributors commit from `<id>+<handle>@users.noreply.github.com` (GitHub's privacy default); the actual bot accounts already match via `\[bot\]`. Author emails are interned in `ChurnResult.author_pool` and per-file maps reference them by `u32` index to keep the bitcode cache compact. Cache version bumped to v2.
- **Feature flag detection**: AST-based detection of feature flag patterns. Three categories: (1) environment variables (`process.env.FEATURE_*` with 12 built-in prefixes), (2) SDK calls (25+ built-in patterns for LaunchDarkly, Statsig, Unleash, GrowthBook, Split, ConfigCat, Flagsmith), (3) config objects (opt-in heuristic for objects named "feature", "flag", "toggle"). Reports flag locations, detection confidence (high/low), guard spans. Cross-references with dead code: flags guarding unused exports are highlighted. Configured via `flags` section in config. Default severity: `off`. CLI: `fallow flags`. Inline suppression: `// fallow-ignore-next-line feature-flag` (line-level) and `// fallow-ignore-file feature-flag` (file-level). The suppression check runs in `crates/cli/src/flags.rs` during both built-in and custom flag collection loops, using `is_suppressed()` / `is_file_suppressed()` with `IssueKind::FeatureFlag`. The file-level check is hoisted outside the per-flag loop for efficiency.
