# Detection capabilities

Non-obvious implementation details for each detection feature. These are NOT discoverable from reading the code alone.

## Parser-level
- **HTML entry files**: `<script src="...">` (module and classic) and `<link rel="stylesheet" href="...">` / `<link rel="modulepreload" href="...">` create graph edges from HTML to referenced assets. Remote URLs skipped. HTML comments stripped before matching. HTML files exempt from unused-file detection.
- **JSX asset references**: lowercase intrinsic `<script src="...">` and `<link rel="stylesheet|modulepreload" href="...">` inside JSX/TSX files create `SideEffect` imports, mirroring the HTML parser. Only `JSXAttributeValue::StringLiteral` values are captured; expression containers (`href={expr}`) and capitalized React-style components (`<Script>`, `<Link>`) are ignored. Needed for SSR frameworks like Hono where layout components emit HTML via JSX. Combined with the web-root-relative resolver branch, `<link href="/static/style.css" />` correctly reaches sibling `static/style.css` files. See issue #105 (till's comment).
- **`html` tagged template asset references**: `` html`<script src="..."></script>` `` and `` html`<link rel="stylesheet|modulepreload" href="..."/>` `` inside any JS/TS file create `SideEffect` imports via the regex-based HTML asset scanner (`crate::html::collect_asset_refs`). Only the bare identifier tag `html` is matched â€” `css`, `sql`, `gql`, `styled.div`, and member/call expressions are deliberately ignored to avoid misreading unrelated tagged templates as HTML. Each template quasi is scanned independently, so an asset reference split across an interpolation boundary (`` html`<script src="${base}/app.js">` ``) is skipped rather than producing a garbled unresolvable specifier. Needed for SSR helpers like `hono/html`, `lit-html`, and `htm` where layout components emit HTML via tagged templates instead of JSX. See issue #105 (till's follow-up comment).
- **JSDoc `import()` type references**: JSDoc comments matching `import('./path').Member` inside `@param`, `@returns`, `@type`, `@typedef`, `@callback`, or other tag contexts are extracted as type-only imports. Supports single/double quotes, bare specifiers, parent-relative paths, union annotations with multiple `import()` expressions per comment, and nested member access (first segment wins). Side-effect-only `import('./types')` without `.Member` is recorded as `SideEffect` to keep the file reachable. Prevents false `unused-export` reports for types referenced only from JSDoc annotations in untyped JS files. Only `/** */` block comments are scanned (single-star `/* */` is not JSDoc). See issue #105 (till's comment).
- **Vue/Svelte SFC**: handles `>` in quoted attributes like `generic="T extends Foo<Bar>"`, `<script src="...">` external script support, HTML comment filtering. Template-visible import tracking: imports used only in markup (`{formatDate(x)}`, `utils.formatDate()`) are credited as used, preventing false unused-import/export reports. Vue credits only `<script setup>` bindings; Svelte excludes `context="module"` scripts from template visibility. Namespace member access in templates (`utils.formatDate`) tracked as member usage.
- **Namespace destructuring**: `const { a, b } = ns` â†’ member accesses. Rest patterns (`const { foo, ...rest } = ns`) â†’ conservative whole-object use. Works with static/dynamic imports and require.
- **Unused import bindings**: via `oxc_semantic` scope-aware symbol analysis. Dead imports don't count as references, improving unused-export precision.
- **TypeScript overload dedup**: `export function foo(): void; export function foo(x: string): string;` treated as single export
- **Import-then-reexport pattern**: `import { X } from './a'; export { X };` is detected as a re-export (semantically equivalent to `export { X } from './a';`) instead of a local export. Without this, the bottom export collides with the original in duplicate-export detection and is never reached by re-export chain propagation. Handles renames on either side (`import { X as Foo }; export { Foo }` or `import { X }; export { X as Y }`), default imports (re-exported with `imported_name: "default"`), and mixed local + re-export in one statement. Order-sensitive: relies on imports appearing before exports in source order. Namespace imports (`import * as ns; export { ns }`) stay as local exports.
- **Class instance members**: `const svc = new MyService(); svc.greet()` and `this.service = new MyService(); this.service.doWork()` track the method as used. Chained `this.field` access uses synthetic `"this.field"` keys in `instance_binding_names`. Factory initializers through wrapper calls are also tracked: `const [svc] = useState(() => new MyService())` binds the first array-destructured element as an instance. Scope-unaware â€” false matches produce false negatives, not false positives.
- **Type-level member access**: `TSQualifiedName` (e.g., `type X = Status.Active`) tracked as member access. Mapped type constraints (`{ [K in Enum]: ... }`, `{ [K in keyof typeof Enum]: ... }`) and `Record<Enum, T>` mark all enum members as used via whole-object use.
- **TypeScript namespace exports**: `export namespace Foo { export function bar() {} }` extracts `Foo` as a single export with inner declarations as `NamespaceMember` entries, not as separate top-level exports. Runtime namespaces (no `declare`) are NOT type-only. `declare namespace`/`declare module` remain type-only. Nested namespaces flatten members into the outermost namespace.
- **Angular component metadata**: Full `@Component`/`@Directive` decorator metadata extraction prevents `unused-class-members` false positives. External templates (`templateUrl`) emit `SideEffect` imports and scan HTML for Angular syntax. Inline templates (`template:` string/template literals) are scanned in-place. `host:` object binding expressions extract referenced member identifiers. `inputs:`/`outputs:` string arrays mark listed members as framework-managed (handles `'name: alias'` syntax). Angular 17+ signal APIs (`input()`, `input.required()`, `output()`, `outputFromObservable()`, `model()`, `viewChild()`, `viewChildren()`, `contentChild()`, `contentChildren()`) set `has_decorator` on the property when the class has an Angular decorator. All template scanning supports `{{ }}`, `[prop]`, `(event)`, `[(ngModel)]`, `*ngIf`, `*ngFor`, `@if`, `@else if`, `@for`, `@switch`, `@defer (when expr)`, `@let name = expr;`, pipes. `*ngFor` and `@for` bindings are tracked as locals. `@let` variables are added as template-local bindings. `@defer` `when`/`prefetch when`/`hydrate when` conditions are scanned for member refs.

## Resolution-level
- **Package.json `exports` subpath**: output dirs (dist/build/out/esm/cjs) mapped back to src/ with source extension fallback, including nested subdirs
- **Pnpm virtual store**: `.pnpm` paths mapped back to workspace source files. Handles injected deps, scoped/unscoped packages, peer dependency suffixes.
- **Package.json `imports` (`#subpath`)**: simple mappings, wildcard patterns, conditional exports. Per-package scoping.
- **React Native platform extensions**: `.web.ts`, `.ios.ts`, `.android.ts`, `.native.ts` resolved alongside standard extensions
- **Web-root-relative paths**: `/src/main.tsx` or `/static/style.css` resolved as a web convention (Vite/Parcel/Hono). Converts to `./src/main.tsx` and resolves from the source file's parent directory first, then falls back to project root. This handles monorepo workspace members where `site/index.html` referencing `/src/main.tsx` resolves to `site/src/main.tsx`, not `<root>/src/main.tsx`. Applied to HTML, JSX, TSX, and JS/TS source files (extensions `.html`, `.jsx`, `.tsx`, `.js`, `.ts`, `.mjs`, `.cjs`, `.mts`, `.cts`). JSX/TSX support is needed for SSR frameworks like Hono where layout components emit `<link href="/static/style.css" />` from JSX templates. See issue #105 (till's comment).
- **CSS scoped package imports**: CSS `@import` with scoped npm packages (`@fontsource/monaspace-neon/400.css`, `@company/design-system/styles.css`) are kept as bare specifiers for node_modules resolution. Unscoped bare imports with CSS/SCSS extensions are normalized to relative paths (`app.css` becomes `./app.css`).
- **Tsconfig path aliases**: per-file discovery resolves `@/utils` by finding nearest tsconfig.json per file
- **Broken tsconfig chain fallback**: when `resolve_file` fails with a tsconfig-loading error (broken `extends` chain, circular extend, self-referencing project reference, malformed JSON, IO error), fall back to `resolve(dir, specifier)` which bypasses `TsconfigDiscovery::Auto`. A solution-style `tsconfig.json` with a broken sibling reference no longer poisons relative/bare-import resolution for files covered by a healthy sibling. A single `tracing::warn!` per unique error surfaces the broken chain to the user. Path aliases in the broken workspace remain unresolvable (they were already broken in `tsc`). See issue #97.
- **SCSS partial resolution**: `@use 'variables'` and `@import '../styles/variables'` resolve to `_variables.scss` per SCSS convention. Extensionless bare specifiers in `.scss`/`.sass` files (excluding `sass:*` built-ins) are normalized to relative paths at extraction time. Resolution fallback tries: (1) `_filename` partial prefix, (2) `specifier/_index` directory index, (3) `specifier/index` plain index. Prevents false unresolved-imports, unlisted-dependencies, and unused-files from SCSS partial references.
- **SCSS include paths**: framework plugins contribute absolute directories to `PluginResult.scss_include_paths`. The Angular plugin reads `projects.*.architect.build.options.stylePreprocessorOptions.includePaths` from `angular.json`; the Nx plugin reads the same key from `project.json`. Nx workspace tokens (`{projectRoot}`, `{workspaceRoot}`) in include paths are expanded before resolution: `{projectRoot}` is computed from the config file's parent relative to the workspace root, `{workspaceRoot}` is stripped (paths are already resolved from workspace root). See issues #103, #114. Bare SCSS specifiers that fail the file-local `try_scss_partial_fallback` retry against each include dir, applying `<name>`, `_<name>`, `<name>/_index`, `<name>/index` candidates. Only triggers for `.scss`/`.sass` importers and normalized bare specifiers (those prefixed with `./` at extraction time) â€” explicit parent-relative paths like `../shared/vars` are left alone.
- **Entry point source fallback for ignored output dirs**: when `package.json` `main`/`module`/`exports` resolves inside an output directory (`dist`, `build`, `out`, `esm`, `cjs`) and the direct `src/` mirror in `try_output_to_source_path` does not exist, `resolve_entry_path` probes `src/index.*`, `src/main.*`, `index.*`, `main.*` in the package root. Prevents the entire `src/` tree from becoming unreachable in libraries whose compiled entry (e.g. `dist/esm2022/index.js`) has no one-to-one source mirror. Emits a single `tracing::info!` per fallback. See issue #102.
- **Workspace package self-reference & cross-workspace fallback**: when a bare specifier's package name matches a workspace package and normal resolution either fails or lands inside `node_modules` for that same package, `try_workspace_package_fallback` strips the package name prefix and resolves the subpath (`button`, `internal/base`, etc.) against a synthetic importer inside the workspace root. Uses `oxc_resolver`'s `resolve_file` so directory indices, source extensions, and the workspace's own `tsconfig.json` aliases still apply. Handles both Node.js v12+ self-referencing (`import '@org/pkg/sub'` from inside `@org/pkg`, common in Angular libraries built with `ng-packagr`) and cross-workspace imports in monorepos that haven't run `npm install`. See issue #106.

## Graph-level
- **Type-only cycle filtering**: `import type` edges carry `is_type_only` through `ImportedSymbol` to cycle detection. Edges where all symbols are type-only are excluded from Tarjan's SCC successor list, preventing false circular dependency reports from type-only bidirectional imports.
- **`export *` chain propagation**: multi-level barrel file chains fully resolved for transitive usage tracking
- **CSS Modules**: default imports (`import styles from '...'`) resolve `styles.className` to named exports via graph-level narrowing. Spread/`Object.values` handled conservatively.

## Analysis-level
- **Duplicate export common-importer filter**: duplicate exports are only reported when at least two files sharing the same export name also share a common importer in the module graph. Unrelated leaf files (e.g., SvelteKit route modules in different directories) that coincidentally export the same name are not flagged.
- **Decorated class members**: members with decorators (NestJS `@Get()`, Angular `@Input()`, TypeORM `@Column()`) not reported unused
- **Configurable class member allowlist**: the built-in Angular/React lifecycle check (see `predicates::lifecycle`) is extended at analysis time with two user-controlled sources: (1) top-level `FallowConfig.used_class_members`, (2) `Plugin::used_class_members()` / `ExternalPluginDef.used_class_members` on active plugins. Both sources feed `AggregatedPluginResult.used_class_members`, which `analyze/mod.rs` merges with the config list into a single `FxHashSet<&str>` passed to `find_unused_members`. Intended for third-party interface patterns where a library calls consumer methods reflectively (ag-Grid's `agInit`/`refresh`, Web Components' `connectedCallback`). The allowlist is scoped to class members â€” matching enum member names are still flagged. See issue #98.
- **Inheritance-aware class members**: `extends` clauses are extracted to build an inheritance map. Parent class `this.*` accesses propagate to child class overrides (e.g., `BaseShape.describe()` calling `this.getArea()` credits `Circle.getArea()`). External member accesses propagate bidirectionally through the hierarchy. Only simple identifier extends are tracked (`class Foo extends Bar`), not mixin patterns. Methods called through parent-typed parameters remain unattributed (requires type inference, out of scope per ADR-001).
- **JSDoc visibility tags** (`@public`, `@internal`, `@beta`, `@alpha`, `@expected-unused`): exports annotated with any of these JSDoc/TSDoc tags are never reported as unused. Stored as a `VisibilityTag` enum on `ExportInfo` and `ExportSymbol`, preserving the semantic distinction (Public = API surface, Internal = sister packages/build tools, Alpha = early preview, Beta = unstable, ExpectedUnused = intentionally unused). Priority when multiple tags present: Public > Internal > Alpha > Beta > ExpectedUnused. Only `/** */` block JSDoc comments recognized. `@api public` TSDoc variant also supported. Word-boundary checked (e.g., `@internalizer` does not match). Survives incremental cache roundtrip via `u8` discriminant encoding. `@expected-unused` differs from other tags: when the tagged export IS actually used (has references from reachable modules), a stale suppression is reported instead of silently accepting it. This supports the pattern of marking API-completeness exports that become adopted over time.
- **Stale suppression detection**: after all detectors run, `SuppressionContext` compares consumed vs. all suppression comments. Unmatched `// fallow-ignore-next-line` and `// fallow-ignore-file` comments are reported as stale. Default severity: `Warn`. CLI-side suppression kinds (complexity, coverage-gaps, feature-flag, code-duplication) are excluded from stale detection to avoid false positives, since their suppression checks run in the CLI layer, not in `find_dead_code_full`. Suppressions for disabled rules (severity `off`) are correctly reported as stale since no detector checks them. The `Suppression` struct stores `comment_line: u32` alongside the target `line: u32` so stale reports point at the comment itself, not the suppressed line. `@expected-unused` JSDoc tags produce stale entries via the unused export detector when the tagged export becomes used. Both comment and JSDoc tag stale entries feed into the same `stale_suppressions` vec on `AnalysisResults` with a discriminated `SuppressionOrigin` enum.
- **Arrow-wrapped dynamic imports**: `React.lazy(() => import('./Foo'))`, `loadable(() => import('./Component'))`, `defineAsyncComponent(() => import('./View'))`, and any function call with an arrow or function expression argument whose body is a sole `import()` expression. The default export of the target module is credited as used (lazy loading wrappers always consume the default export). Handles expression body (`() => import('./x')`), block body (`() => { return import('./x'); }`), and function expressions (`function() { return import('./x'); }`). The import span is added to `handled_import_spans` to prevent duplicate side-effect edges from the generic `visit_import_expression` walker.
- **Dynamic import `.then()` callback**: `import('./lib').then(m => m.foo)` tracks `foo` as a named import from `./lib`. The standard Angular lazy-loading pattern (`loadComponent: () => import('./x').then(m => m.Component)`). Three sub-patterns: (1) expression body member access (`m => m.foo`) extracts specific member names as `destructured_names`, (2) destructured parameter (`({ foo, bar }) => { ... }`) extracts names from the object pattern, (3) identifier parameter with block body (`m => { ... }`) registers as a namespace binding for member-access narrowing. Also handles `({ default: m.Foo })` object literal bodies (React.lazy `.then` pattern) and function expression callbacks. The import span is added to `handled_import_spans` to prevent duplicate side-effect edges. See issue #115.
- **Infrastructure entry points**: Dockerfiles, Procfiles, fly.toml scanned for source file refs. Searches root and config/docker/deploy subdirs.
- **TypeScript project references**: tsconfig.json `references` field discovered as workspaces (additive with npm/pnpm workspaces)
- **CI file scanning**: `.gitlab-ci.yml` and `.github/workflows/*.yml` scanned for binary invocations (npx, direct binaries). Maps binaries to package names to prevent false "unused dependency" reports for CI-only packages.
- **Production mode**: excludes test/dev files, only start/build scripts, skips unused devDeps, detects type-only deps. Config file exclusion (`*.config.*`) is root-anchored only; nested `*.config.*` files (e.g., Angular's `src/app/app.config.ts`) are preserved. All production exclude globs use `literal_separator(true)` so `*` cannot cross path separators. See issue #111.
- **Test-only dependency detection**: `find_test_only_dependencies` identifies production dependencies imported exclusively from test/dev/config files (should be devDependencies). Uses a dual classification: glob patterns from `PRODUCTION_EXCLUDE_PATTERNS` for test/story files (`**/*.test.*`, `**/*.spec.*`, `**/__tests__/**`, etc.) and the curated `is_config_file()` predicate for tooling config files at any depth. The predicate approach avoids false matches on application config files like `app.config.ts` while correctly matching workspace-level configs like `packages/foo/vitest.config.ts`. See issues #111, #112.
- **Platform built-in modules**: `node:`, `bun:`, `cloudflare:`, `sass:` prefixed imports and Deno `std` are recognized as builtins â€” never flagged as unlisted dependencies. The `sass:` prefix covers the Sass compiler's built-in modules (`sass:math`, `sass:string`, `sass:color`, `sass:list`, `sass:map`, `sass:meta`, `sass:selector`) imported via SCSS `@use 'sass:math'`. See issue #104.
- **`ignoreDependencies` dual suppression**: listed packages are excluded from both unused dependency AND unlisted dependency detection, making it useful for runtime-provided packages (e.g., `bun:sqlite`).
- **@types/ unlisted dep suppression**: imports from `pkg` not flagged as unlisted when `@types/pkg` is in deps (TypeScript resolves types from `@types/` and erases the import regardless of `import type` syntax). Scoped packages use DefinitelyTyped convention (`@scope/pkg` â†’ `@types/scope__pkg`).
- **Architecture boundary violations**: user-defined zones (glob patterns) with directional import rules. Fires at direct import site using resolved target's zone. Unzoned files are unrestricted in both directions. First-match zone classification. Self-imports always allowed. Inline and file-level suppression supported. Built-in presets: `layered` (presentation/application/domain/infrastructure), `hexagonal` (adapters/ports/domain), `feature-sliced` (app/pages/widgets/features/entities/shared), `bulletproof` (app/features/shared/server â€” shared zone covers components, hooks, lib, utils, utilities, providers, shared, types, styles, i18n). Presets auto-detect `rootDir` from tsconfig.json for pattern prefix. User zones/rules merge on top of presets (same-name replaces). Zones with zero file matches emit warnings.
- **Static test coverage gaps**: module graph reachability from test entry points to runtime files/exports. Not line-level coverage. Test files are identified by plugin patterns and production-mode exclusion rules. Reports runtime files with no test dependency path and runtime exports with no test-reachable reference chain. Gated by `--coverage-gaps` flag on the health command and `coverage-gaps` rule severity.
- **Complexity suppression**: health command complexity findings (functions exceeding cyclomatic/cognitive thresholds) support inline suppression via `// fallow-ignore-next-line complexity` and `// fallow-ignore-file complexity`. The blanket `// fallow-ignore-next-line` (no kind) also suppresses complexity findings on the next line. The check happens in `collect_findings()` in `crates/cli/src/health/mod.rs` using `suppress::is_suppressed()` with `IssueKind::Complexity` (discriminant 18). File scores and vital signs are NOT affected by suppressions (they reflect actual complexity, not alerting). See issue #108 (till's comment).
- **Script multiplexer handling**: `concurrently`, `npm-run-all`, `npm-run-all2`, `run-s`, `run-p`, `run-s2`, `run-p2` are recognized as script multiplexers. Their positional arguments are treated as script names (not binary names or file paths) and excluded from `used_packages`. Only the multiplexer binary itself is recorded. This prevents false "unused dependency" reports for packages referenced only via `npm:` prefixed script delegation (e.g., `concurrently "npm:dev" "npm:test"`).
- **Entry export validation** (`--include-entry-exports`): when enabled, entry file exports (package.json main/module/exports, framework convention files) are subject to unused-export detection instead of being automatically marked as used. Catches typos in framework exports (e.g., `meatdata` instead of `metadata` in Next.js pages). Still respects plugin `used_exports` rules for partial checking. CLI-only flag, not in config schema.
- **Single-file filtering** (`--file`): scopes check output to specified file(s). Full graph still builds (correct results require cross-file analysis), but only issues in matching files are reported. Dependency-level issues (unused deps, dev deps, type-only deps) are suppressed in file mode since they are project-wide properties. Warns when a path is not found. Useful for lint-staged pre-commit hooks.
- **Feature flag detection**: AST-based detection of feature flag patterns. Three categories: (1) environment variables (`process.env.FEATURE_*` with 12 built-in prefixes), (2) SDK calls (25+ built-in patterns for LaunchDarkly, Statsig, Unleash, GrowthBook, Split, ConfigCat, Flagsmith), (3) config objects (opt-in heuristic for objects named "feature", "flag", "toggle"). Reports flag locations, detection confidence (high/low), guard spans. Cross-references with dead code: flags guarding unused exports are highlighted. Configured via `flags` section in config. Default severity: `off`. CLI: `fallow flags`. Inline suppression: `// fallow-ignore-next-line feature-flag` (line-level) and `// fallow-ignore-file feature-flag` (file-level). The suppression check runs in `crates/cli/src/flags.rs` during both built-in and custom flag collection loops, using `is_suppressed()` / `is_file_suppressed()` with `IssueKind::FeatureFlag`. The file-level check is hoisted outside the per-flag loop for efficiency.
